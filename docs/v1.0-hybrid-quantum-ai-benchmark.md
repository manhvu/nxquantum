# v1.0 Hybrid Quantum-AI Benchmark Spec

Status note (as of March 29, 2026):

1. This benchmark specification is active for Phase 20+ execution.
2. Benchmark scripts are implemented in `bench/hybrid_quantum_ai_*.exs` and `bench/turboquant_rerank_benchmark.exs`.

## Goal

Define reproducible, migration-relevant benchmark scenarios for hybrid quantum-AI workflows with explicit classical baselines.

## Scope (Phase 20)

Required scenarios:

1. `rerank_quality_delta`
2. `rerank_quality_delta_turboquant`
3. `constrained_optimization_assistant`
4. `latency_fallback_impact`

Required outputs:

1. quality metrics with baseline deltas,
2. latency/cost/fallback metrics,
3. machine-readable report artifact.

## Benchmark Inputs

## Dataset Catalog (Versioned)

All datasets must be version-pinned and loaded via explicit manifest metadata.

1. Rerank dataset (`rq_small_v1` for CI, `rq_medium_v1` for local/dev):
   - `dataset_id`: `rq_small_v1` / `rq_medium_v1`
   - canonical files:
     - `bench/datasets/rerank/rq_small_v1.csv`
     - `bench/datasets/rerank/rq_medium_v1.csv`
   - row schema:
     - `query_id: string`
     - `candidate_id: string`
     - `classical_score: float`
     - `label: 0 | 1 | 2 | 3`
     - `query_embedding: pipe-delimited floats`
     - `candidate_embedding: pipe-delimited floats`

2. Constrained optimization dataset (`co_knapsack_v1`):
   - canonical file (planned):
     - `bench/datasets/constrained/co_knapsack_v1.jsonl`
   - row schema:
     - `problem_id: string`
     - `weights: [number]`
     - `values: [number]`
     - `capacity: number`
     - `known_best_value: number`

3. Latency/fallback workload manifest (`lf_mixed_v1`):
   - canonical file (planned):
     - `bench/datasets/latency/lf_mixed_v1.jsonl`
   - row schema:
     - `workload_id: string`
     - `tool_name: string`
     - `input_ref: string`
     - `transport_mode: fixture | live_smoke | live`
     - `expected_policy: strict | allow_classical_fallback`

Transport mode semantics for this benchmark:

1. `fixture`: deterministic baseline lane.
2. `live_smoke`: readiness lane for env/config and contract-path validation; do not treat as full remote execution evidence.
3. `live`: remote execution lane for provider lifecycle behavior and probabilistic outcome analysis.

## Determinism Rules

1. Every run must include `seed`, `dataset_id`, `dataset_sha256`, `scenario_id`, and `runtime_profile`.
2. Train/validation/test splits must be fixed and versioned.
3. Any schema change requires version bump (`*_v2`) and migration notes.

## Scenario Definitions

## Scenario A: `rerank_quality_delta`

Objective:

1. Measure ranking quality delta between:
   - classical-only reranker,
   - hybrid reranker through `quantum_kernel_rerank.v1`.

Minimum metrics:

1. `ndcg_at_10`
2. `map_at_10`
3. `latency_p50_ms`, `latency_p95_ms`
4. `fallback_rate`
5. `memory_bytes_per_vector`

Acceptance guidance (initial):

1. quality delta must be reported against baseline with confidence interval,
2. no claim without caveat and baseline reference.

## Scenario B: `constrained_optimization_assistant`

Objective:

1. Compare optimization quality/feasibility between:
   - classical heuristic baseline,
   - hybrid path via `constrained_optimize.v1`.

Minimum metrics:

1. `feasible_solution_rate`
2. `objective_gap_to_known_best`
3. `iterations_to_first_feasible`
4. `latency_p50_ms`, `latency_p95_ms`

## Scenario C: `latency_fallback_impact`

Objective:

1. Quantify operational behavior under mixed workloads.

Minimum metrics:

1. `request_success_rate`
2. `fallback_rate`
3. `typed_error_rate_by_code`
4. `latency_p50_ms`, `latency_p95_ms`

Breakdowns required:

1. by tool (`quantum_kernel_rerank.v1`, `constrained_optimize.v1`)
2. by transport mode (`fixture`, `live_smoke`, `live`)
3. by runtime profile.
