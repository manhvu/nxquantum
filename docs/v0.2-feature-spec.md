# NxQuantum v0.2 Feature Specification

## Release Theme

Nx-native differentiable quantum workflows for practical ML research and production prototyping.

## Product Outcomes

1. Make hybrid classical-quantum models first-class in Elixir ML stacks.
2. Deliver reproducible, backend-aware execution without Python dependency chains.
3. Provide a clean API that scales from toy circuits to meaningful research experiments.

## Target Users

1. ML researchers experimenting with variational circuits and hybrid neural networks.
2. Data engineers needing deterministic, backend-accelerated numerical pipelines.
3. Elixir/Nx users who want quantum primitives integrated into existing BEAM workflows.

## Design Principles

1. Pipelines over configuration.
2. Tensor operations first (`Nx.Defn` compatibility by default).
3. Deterministic behavior as a core contract.
4. Explicit runtime profiles instead of hidden backend magic.
5. Domain clarity through DDD + Hexagonal boundaries.

## Non-Goals (v0.2)

1. Full noisy hardware calibration stack.
2. Large-scale fault-tolerant simulation.
3. Broad quantum hardware provider SDK integrations.

## Proposed Public API (v0.2)

```elixir
alias NxQuantum.Circuit
alias NxQuantum.Gates
alias NxQuantum.Estimator
alias NxQuantum.Runtime
alias NxQuantum.Grad

circuit =
  Circuit.new(qubits: 2)
  |> Gates.h(0)
  |> Gates.ry(0, theta: Nx.tensor(0.2))
  |> Gates.cnot(control: 0, target: 1)

profile = Runtime.profile!(:cpu_compiled)

expectation =
  Estimator.expectation(
    circuit,
    observable: :pauli_z,
    wire: 1,
    runtime_profile: profile
  )

{loss, grads} =
  Grad.value_and_grad(fn theta ->
    circuit
    |> Circuit.bind(theta: theta)
    |> Estimator.expectation(observable: :pauli_z, wire: 1, runtime_profile: profile)
    |> Nx.subtract(1.0)
    |> Nx.power(2)
  end, Nx.tensor(0.2))
```

## Runtime Profiles (Pareto Coverage)

1. `:cpu_portable` -> `Nx.BinaryBackend` (P0).
2. `:cpu_compiled` -> `EXLA.Backend` host CPU (P0).
3. `:nvidia_gpu_compiled` -> `EXLA.Backend` CUDA (P0).
4. `:torch_interop_runtime` -> `Torchx.Backend` (P1).

Deterministic policy contract:

1. Fallback policy `:strict` returns typed error.
2. Fallback policy `:allow_cpu_compiled` downgrades from unavailable accelerated profile to `:cpu_compiled` with warning code.

## Feature Set

### F1. Deterministic Core Simulator (P0)

Scope:

1. State-vector evolution for 1-2 qubit correctness baseline.
2. Pauli expectation for `:pauli_x | :pauli_y | :pauli_z`.
3. Deterministic tolerance-driven numeric comparisons.

Acceptance:

1. Analytical references match for canonical circuits.
2. Norm preservation property tests pass.

### F2. Hybrid Training and Autodiff (P0)

Scope:

1. `Nx.grad` compatible expectation path.
2. Deterministic one-step optimization scenarios.
3. Axon-compatible quantum layer boundary.

Acceptance:

1. Deterministic SGD reference scenario passes.
2. Same-seed runs are byte-identical for init, grads, and updated params.

### F3. Differentiation Modes (P1)

Scope:

1. `:backprop` as default mode.
2. `:parameter_shift` for gate-parameterized circuits.
3. `:adjoint` as high-performance simulator gradient mode for supported gate sets.
4. Explicit `circuit_builder` contract for `:adjoint` to keep gradient semantics deterministic and inspectable.

Acceptance:

1. Gradient consistency checks across modes within tolerance.
2. Unsupported mode/gate combos return typed errors.
3. Missing `circuit_builder` in `:adjoint` returns typed contract error.

Adjoint mode contract (v0.2):

1. Supported gates: `:h`, `:x`, `:y`, `:z`, `:rx`, `:ry`, `:rz`, `:cnot`.
2. Parameter mapping: differentiable rotation gates are mapped to flattened parameter indices by operation order.
3. Unsupported operation returns error code `:unsupported_gradient_mode` and includes operation name.
4. Missing builder returns error code `:adjoint_requires_circuit_builder`.

### F4. Shots and Noise Channels (P1)

Scope:

1. Deterministic seeded shot sampling.
2. Optional expectation estimation by finite shots.
3. Initial noise channels: depolarizing + amplitude damping.

Acceptance:

1. Infinite-shot approximation converges toward analytic expectation.
2. Same seed + same shots yields identical sampled results.

### F5. Circuit Optimization Pipeline (P1)

Scope:

1. Canonical pass pipeline (`simplify`, `fuse`, `cancel`).
2. Preserve semantics under optimization.
3. Expose optimization report (before/after gate counts).

Acceptance:

1. Optimized and unoptimized expectations match within tolerance.
2. Gate count reduction observed on benchmark circuits.

### F6. Quantum Kernel Methods (P2)

Scope:

1. Feature-map circuits for kernel generation.
2. Kernel matrix generation API.
3. Interop with classical ML model training workflows.

Acceptance:

1. Kernel matrix is symmetric positive semidefinite within tolerance.
2. Deterministic seeded matrix generation.

1. Baseline benchmark set for 1-12 qubit circuits across P0 profiles.
2. Batch execution benchmark (`batch_size` 1, 8, 32, 128).
3. Track compile vs run time split for EXLA profiles.

## DX and Adoption Requirements

1. One-command reproducible setup via `mise install`.
2. Single canonical project workflow through `mix`.
3. Cookbook-style examples for Axon integration and kernel workflows.
4. Clear typed errors and remediation messages for runtime-profile issues.

## Risks and Mitigations

1. Risk: numerical drift across backends.
   Mitigation: shared deterministic reference suite with per-backend tolerances.
2. Risk: API fragmentation as advanced features grow.
   Mitigation: keep `Circuit`, `Estimator`, and `Runtime` as stable facade modules.
3. Risk: complexity from too many differentiation options.
   Mitigation: default to `:backprop`; mark advanced modes as explicit opt-in.

## Delivery Phases

1. Milestone A (P0): F1 + F2 + runtime profile contract stabilization.
2. Milestone B (P1): F3 + F4 + F5.
3. Milestone C (P2): F6 + ecosystem polish and examples.

## Milestone A (P0) Evidence Checklist

Required executable evidence before moving to Milestone B:

1. Deterministic 1-2 qubit state-vector evolution is validated with analytical feature scenarios.
2. Pauli expectation values (`:pauli_x`, `:pauli_y`, `:pauli_z`) are covered by deterministic references.
3. Property tests cover both:
   1. gate-composition invariants,
   2. state normalization invariants.
4. Runtime profile resolution and fallback policy behavior are validated through executable feature scenarios.
5. Hybrid training deterministic behavior (fixed-seed reproducibility and one-step update determinism) is executable in the feature suite.

## Milestone B (P1) Evidence Checklist

Required executable evidence before moving to Milestone C:

1. Gate application and expectation calculation paths are executed through explicit `Nx.Defn` kernels.
2. Gradient consistency across `:backprop`, `:parameter_shift`, and `:adjoint` is verified within deterministic tolerance.
3. Seeded shot sampling reproducibility is validated in executable acceptance scenarios.
4. Initial noise channels include depolarizing and amplitude damping with deterministic behavioral assertions.
5. Optimization passes (`simplify`, `fuse`, `cancel`) preserve semantics and expose deterministic gate-count report metadata.
6. Benchmark scripts and a baseline report are committed under `bench/` with a reproducible command.

## Milestone C (P2) Evidence Checklist

Required executable evidence before moving to Phase 4:

1. `NxQuantum.Kernels.matrix/2` supports deterministic seed-aware kernel matrix generation.
2. Kernel validity checks (shape, symmetry, PSD approximation, reproducibility) are covered by executable tests/features.
3. Axon integration workflow has end-to-end documented examples for deterministic train-step usage.
4. Additional recipe/tutorial docs are included and discoverable from the README.

## Feature Status Legend

- `planned`: specified but not scaffolded in code.
- `scaffold`: module/function skeleton exists, behavior incomplete.
- `implemented`: functional behavior available.
- `verified`: implemented with deterministic acceptance and property coverage.
