# NxQuantum v0.5 Feature Specification

## Release Theme

Production-grade provider bridges for `IBM Runtime`, `AWS Braket`, and `Azure Quantum`.

## Why v0.5 Matters

1. Close the largest documented gap versus Python-first stacks: hardware-provider depth.
2. Convert existing provider bridge foundations into real provider integrations with typed deterministic contracts.
3. Reduce migration friction for teams moving Qiskit/Braket/Azure workflows into BEAM-native services.

## Priority Selection (Pareto)

Selected for v0.5 (highest impact per integration effort):

1. **P0: IBM Runtime + AWS Braket adapters** (two integrations cover most migration demand and immediate hardware breadth).
2. **P1: Azure Quantum adapter + cross-provider normalization**.
3. **P2: Provider-specific migration packs, benchmark evidence, and support-tier hardening**.

Deferred post-v0.5:

1. Direct vendor adapters where broker coverage is sufficient (for example direct IonQ/Pasqal/Quantinuum APIs).
2. Analog-Hamiltonian-specific provider workflows as first-class runtime paths.
3. Automatic multi-provider failover/reroute behavior (explicit provider selection remains required).

## Top-3 Provider Selection Rationale

1. `IBM Runtime`: strongest alignment with primitive-centric workflows (`Estimator`/`Sampler`) and Qiskit migration paths.
2. `AWS Braket`: broad multi-provider hardware access behind one API family with explicit task lifecycle contracts.
3. `Azure Quantum`: additional multi-provider coverage and enterprise platform integration.

## Feature Set

### F1. Provider Capability Contract v1 [P0]

Goal:

1. Expose an explicit typed capability envelope so orchestration can reject unsupported paths deterministically.

Scope:

1. Provider capability map schema:
   - `supports_estimator`
   - `supports_sampler`
   - `supports_batch`
   - `supports_dynamic`
   - `supports_cancel_in_running`
   - `supports_calibration_payload`
2. Typed target metadata envelope:
   - `provider`
   - `target_id`
   - `target_class`
   - `region`
3. Deterministic unsupported-capability diagnostics.

Acceptance:

1. Capability checks occur before remote submission.
2. Unsupported requests return typed errors with provider/target context.
3. Capability schema is stable across all top-3 adapters.

### F2. IBM Runtime Provider Adapter [P0]

Goal:

1. Execute primitive-centric workflows via IBM Runtime through stable NxQuantum contracts.

Scope:

1. `NxQuantum.Adapters.Providers.IBMRuntime` implementation behind `NxQuantum.Ports.Provider`.
2. Config envelope:
   - auth token
   - account channel/context metadata
   - target/backend selector
3. Lifecycle operations:
   - `submit` -> create primitive execution job
   - `poll` -> map job status into typed state envelope
   - `cancel` -> map cancellation behavior
   - `fetch_result` -> normalize result payload
4. Typed provider error mapping:
   - transport/auth
   - invalid backend/capability
   - provider runtime failure
5. Primitive payload mapping for estimation and sampling workloads.

Acceptance:

1. IBM workflow can run through `ProviderBridge.run_lifecycle/3` with deterministic typed outputs for fixture-backed scenarios.
2. Job statuses map into stable NxQuantum lifecycle states.
3. Auth/config/provider failures map into typed diagnostics with provider identifiers.

### F3. AWS Braket Provider Adapter [P0]

Goal:

1. Enable hardware execution over Braket with deterministic lifecycle and result normalization.

Scope:

1. `NxQuantum.Adapters.Providers.AwsBraket` implementation behind `NxQuantum.Ports.Provider`.
2. Config envelope:
   - region
   - credentials/profile
   - device ARN or target alias
3. Lifecycle mapping:
   - task submission
   - task status polling
   - task cancellation
   - task result retrieval
4. Status normalization for Braket task lifecycle states.
5. Gate-model execution path as v0.5 primary scope.
6. Typed diagnostics for quota/authorization/invalid target and terminal failed states.

Acceptance:

1. Braket adapter supports deterministic lifecycle contract behavior via fixtures.
2. Braket-specific statuses map into typed normalized states with provider metadata.
3. Unsupported target classes return typed, deterministic preflight errors.

### F4. Azure Quantum Provider Adapter [P1]

Goal:

1. Add third provider bridge with explicit target/provider semantics and typed lifecycle behavior.

Scope:

1. `NxQuantum.Adapters.Providers.AzureQuantum` implementation behind `NxQuantum.Ports.Provider`.
2. Config envelope:
   - workspace identifiers
   - auth context
   - target/provider selector
3. Lifecycle mapping:
   - submit
   - poll
   - cancel
   - fetch_result
4. Provider-specific cancellation and terminal-state caveat metadata.
5. Result envelope normalization for gate-model workflow outputs.

Acceptance:

1. Azure job lifecycle is available behind the same `ProviderBridge` contract.
2. Provider-specific behavior differences are surfaced via typed metadata, not hidden.
3. Cross-provider scenario parity (for equivalent intents) passes deterministic contract tests.

### F5. Cross-Provider Contract Equivalence [P1]

Goal:

1. Ensure portability where possible and explicit boundaries where not possible.

Scope:

1. Common normalized job envelope (`id`, `state`, `provider`, `target`, `submitted_at`, `metadata`).
2. Cross-provider error taxonomy and mapping table.
3. Capability-driven preflight checks at facade boundary.
4. Deterministic redaction policy for provider credentials in error metadata and logs.

Acceptance:

1. Equivalent workflows produce consistent typed contract structure across top-3 adapters.
2. Provider-specific non-portable behavior is explicit and test-covered.
3. No silent provider fallback occurs when requested provider is unavailable.

### F6. Migration and Enablement Assets [P2]

Goal:

1. Make provider migration practical for production teams.

Scope:

1. Provider-specific migration packs:
   - Qiskit Runtime -> NxQuantum
   - Braket workflows -> NxQuantum
   - Azure Quantum workflows -> NxQuantum
2. Runnable tutorial packs for estimation/sampling/transpilation-constrained workloads.
3. Benchmark matrix with reproducible scripts and provider caveat disclosures.
4. Support-tier policy and release-note templates.

Acceptance:

1. Every migration pack maps to at least one executable acceptance scenario.
2. Tutorials run with deterministic fixture paths even when remote credentials are unavailable.
3. Positioning claims are backed by reproducible benchmark evidence.

### F7. OpenTelemetry Observability Standard [P1]

Goal:

1. Deliver first-class trace/log/metric observability for provider and QML workflow operations.

Scope:

1. OpenTelemetry instrumentation for provider lifecycle spans:
   - `nxq.provider.submit`
   - `nxq.provider.poll`
   - `nxq.provider.cancel`
   - `nxq.provider.fetch_result`
2. Workflow and pipeline spans:
   - `nxq.workflow.run`
   - `nxq.transpile.run`
   - `nxq.mitigation.pipeline`
3. Visibility profiles:
   - `high_level`
   - `granular`
   - `forensics`
4. Core metrics for all top-3 providers:
   - latency histograms
   - queue and execution timing
   - error and success counters
5. Deterministic experiment fingerprint and portability-delta telemetry contracts.
6. Deterministic redaction and bounded-cardinality telemetry safeguards.

Acceptance:

1. Profile changes alter telemetry detail without breaking base contract shape.
2. Core traces/logs/metrics are emitted for equivalent workflows across top-3 providers.
3. Fingerprint/portability telemetry is deterministic for identical workflow inputs.
4. Telemetry contracts are schema-tested and documented.

## Proposed API/Contract Additions (v0.5)

```elixir
alias NxQuantum.ProviderBridge

provider = NxQuantum.Adapters.Providers.IBMRuntime

payload = %{
  workflow: :estimator,
  circuit: circuit,
  observable: :pauli_z,
  wire: 0,
  shots: 4096
}

{:ok, lifecycle} =
  ProviderBridge.run_lifecycle(provider, payload,
    target: "ibm_backend_name",
    capability_contract: :v1
  )
```

```elixir
{:error, %{code: :unsupported_provider_capability, capability: :supports_dynamic, provider: :aws_braket}} =
  ProviderBridge.submit_job(NxQuantum.Adapters.Providers.AwsBraket, payload, dynamic: true)
```

## Scenario Catalog (v0.5)

### S1: Capability preflight rejects unsupported dynamic request

1. Given provider capabilities do not include dynamic execution
2. When a dynamic workflow is submitted
3. Then `unsupported_provider_capability` is returned with provider/target metadata

### S2: IBM Runtime lifecycle contract behavior

1. Given valid IBM Runtime provider credentials and target backend
2. When estimator workflow is submitted and polled to completion
3. Then `fetch_result` returns normalized typed payload with provider metadata

### S3: AWS Braket lifecycle and terminal status mapping

1. Given valid Braket device selection and region config
2. When a gate-model task is submitted and reaches terminal state
3. Then normalized state and typed result/error metadata are deterministic

### S4: Azure cancellation caveat surfaced explicitly

1. Given an Azure target where cancellation behavior has provider caveats
2. When a running job is cancelled
3. Then result includes typed cancellation caveat metadata

### S5: Cross-provider typed envelope equivalence

1. Given equivalent sampling intents for IBM, Braket, and Azure
2. When each provider returns terminal results
3. Then normalized contract fields are consistent and provider-specific extras are nested under metadata

### S6: Credential/config redaction

1. Given invalid provider credentials
2. When provider submission fails
3. Then returned diagnostics are typed and redact secrets deterministically

### S7: High-level observability profile emits core telemetry

1. Given observability profile is `high_level`
2. When provider workflow runs to terminal state
3. Then mandatory lifecycle spans and core latency/error metrics are emitted

### S8: Granular profile emits lifecycle breakdown telemetry

1. Given observability profile is `granular`
2. When provider workflow includes queue and execution phases
3. Then queue wait and execution metrics are emitted separately with stable schema keys

### S9: Fingerprint + portability delta telemetry is deterministic

1. Given equivalent workloads are executed across top-3 providers
2. When observability is enabled with portability telemetry
3. Then `experiment_fingerprint` and portability-delta contracts are emitted deterministically

2. IBM and Braket adapter implementations.
3. Fixture-backed lifecycle and error contract scenarios.

### Milestone J (P1): Azure + cross-provider normalization

1. Azure adapter implementation.
2. Cross-provider state/error/result normalization.
3. Provider caveat metadata model and tests.

1. Migration packs and runnable tutorials.
2. Benchmark evidence and reproducible scripts.

## Success Metrics

1. Top-3 provider adapters pass lifecycle contract suite with deterministic fixtures.
2. `submit/poll/cancel/fetch_result` contract remains stable across providers.
3. Migration docs reduce provider-specific glue requirements for targeted workflows.
4. v0.5 release evidence includes reproducible benchmark and typed failure catalogs.
5. OpenTelemetry observability contracts are stable across visibility profiles.

## Risks and Mitigations

1. Risk: provider API drift and SDK/runtime version churn.
   Mitigation: adapter boundary tests + recorded fixtures + version pinning policy.
2. Risk: hidden non-portable behavior across providers.
   Mitigation: capability preflight + explicit provider caveat metadata.
3. Risk: accidental silent fallback/reroute causing nondeterministic behavior.
   Mitigation: strict explicit provider selection and typed errors for unavailable capability.
