## Feature Acceptance Criteria

### F1) Provider Adapter Transport Readiness

Required outcomes:

1. Fixture-first deterministic behavior remains default and stable for CI.
2. Adapter transport seams support optional live smoke lanes without public API changes.
3. Lifecycle normalization and typed error taxonomy remain contract-stable.
4. Unknown provider responses never silently coerce into valid states.

Acceptance checklist:

1. Fixture lane tests pass deterministically for top-3 adapters.
2. Optional live smoke-lane configuration path is documented and guarded by env/config toggles.
3. `submit/poll/cancel/fetch_result` shape and typed error contracts are unchanged.
4. Invalid response/state paths return `provider_invalid_response` or `provider_invalid_state` deterministically.

### F2) Provider Scenario Contract Completion

Required outcomes:

1. Remaining scaffold-only provider feature step modules are converted into executable assertions.
2. Scenario behavior includes both success and representative failure assertions.
3. Deterministic metadata and typed failure behavior are validated for each scenario family.

Acceptance checklist:

1. The following step modules contain executable assertion logic (no scaffold capture-only behavior):
   - `test/features/steps/provider_dynamic_circuit_capabilities_steps.ex`
   - `test/features/steps/provider_mitigation_calibration_contracts_steps.ex`
   - `test/features/steps/provider_topology_execution_policies_steps.ex`
   - `test/features/steps/provider_simulation_strategy_fallback_steps.ex`
   - `test/features/steps/provider_batched_primitives_performance_steps.ex`
2. Feature suite executes all corresponding `.feature` files with assertions, not no-op step accumulation.
3. Unit/property tests exist for core invariants behind each scenario family.

### F3) Documentation and Evidence Consistency

Required outcomes:

1. v0.5 and v0.6 status language is aligned across roadmap/spec/acceptance docs.
2. Fixture vs optional live smoke-lane scope is explicit in docs.
3. Release evidence references are reproducible and link-valid.

Acceptance checklist:

1. No stale “pending/planned” wording remains for completed v0.5 implementation paths.
2. v0.6 docs clearly declare boundary and non-goals.
3. Docs build includes all v0.6 artifacts and passes link integrity checks.

## v0.6 Milestone Gates (Acceptance-Driven)

### Milestone L (P0): Transport and Smoke-Lane Readiness

1. F1 acceptance criteria are approved and mapped to executable checks.
2. No public API contract drift is introduced by adapter seam hardening.

### Milestone M (P1): Scenario Completion
