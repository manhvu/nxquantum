# v0.5 Provider Implementation Plan (Top 3)

## Objective

Deliver production-grade provider support for:

1. `IBM Runtime`
2. `AWS Braket`
3. `Azure Quantum`

while preserving NxQuantum's deterministic typed contract model and hexagonal boundaries.

Architecture references:

1. `docs/adr/0002-provider-capability-contract-v1.md`
2. `docs/adr/0003-ibm-runtime-provider-adapter.md`
3. `docs/adr/0004-aws-braket-provider-adapter.md`
4. `docs/adr/0005-azure-quantum-provider-adapter.md`
5. `docs/adr/0006-opentelemetry-observability-standard.md`

## Scope and Non-Goals

In scope (v0.5):

1. Provider adapter implementations behind `NxQuantum.Ports.Provider`.
2. Deterministic `submit`/`poll`/`cancel`/`fetch_result` behavior with typed error mapping.
3. Capability preflight and provider-specific caveat metadata.
4. Fixture-backed deterministic contract suite and optional credentialed smoke lanes.
5. OpenTelemetry traces/logs/metrics with profile-driven visibility.

Out of scope (v0.5):

1. Automatic cross-provider reroute/failover.
2. Direct vendor adapters when broker coverage already exists.
3. Analog-Hamiltonian-native runtime as first-class execution path.
4. Mandatory third-party telemetry backend selection (OpenTelemetry remains optional/configurable).

## Architecture Constraints (must hold)

1. `Domain <- Application <- Adapters` dependency direction remains intact.
2. Facade APIs (`Estimator`, `Sampler`, `ProviderBridge`) remain stable unless explicitly versioned.
3. Provider-specific payloads are normalized at adapter boundaries; they must not leak into domain types.
4. All remote variability becomes typed metadata, not ad-hoc string handling.
5. Observability overhead must be profile-bounded and deterministic in contract behavior.

## Provider-Specific Knowledge to Encode

### IBM Runtime

1. Primitive-centric workflows (`Sampler`/`Estimator`) are first-class.
2. Job lifecycle includes status polling and cancellation APIs.
3. Runtime option surfaces and backend targeting require explicit configuration contracts.

Implementation implications:

1. IBM adapter should map NxQuantum primitive intents directly to runtime primitive payloads.
2. Capability model must encode primitive support and backend constraints.
3. Error taxonomy needs robust mapping for auth, backend unavailability, and runtime execution failures.

### AWS Braket

1. Task lifecycle is explicit with submit/get/cancel primitives.
2. Device targeting is explicit (for example by device ARN).
3. Provider/device diversity is high; gate-model subset should be the v0.5 primary scope.

Implementation implications:

1. Adapter must model Braket task state transitions and terminal state semantics.
2. Capability preflight must reject unsupported target/device classes deterministically.
3. Result normalization must shield consumers from provider-variant payload shape differences.

### Azure Quantum

1. Workspace/target/provider selection is explicit.
2. Job lifecycle is available but cancellation semantics can vary by provider/phase.
3. Target availability and provider mix are broader than a single vendor path.

Implementation implications:

1. Adapter must attach provider/target caveat metadata to lifecycle transitions.
2. Cancellation behavior must be represented as typed metadata, not assumed uniform.
3. Result normalization contract must separate common fields from provider-specific extras.

## Contract Design v1 (Provider Bridge)

### Normalized Job Envelope

```elixir
%{
  id: String.t(),
  state: :submitted | :queued | :running | :completed | :cancelled | :failed,
  provider: atom() | String.t(),
  target: String.t(),
  submitted_at: String.t() | nil,
  metadata: map()
}
```

### Normalized Result Envelope

```elixir
%{
  job_id: String.t(),
  state: :completed | :cancelled | :failed,
  provider: atom() | String.t(),
  target: String.t(),
  payload: map(),
  metadata: map()
}
```

### Normalized Error Envelope

```elixir
%{
  code:
    :provider_transport_error
    | :provider_auth_error
    | :provider_invalid_state
    | :provider_invalid_response
    | :provider_capability_mismatch
    | :provider_execution_error
    | :provider_rate_limited,
  provider: atom() | String.t(),
  operation: :submit | :poll | :cancel | :fetch_result,
  reason: term(),
  metadata: map()
}
```

### Capability Contract

```elixir
%{
  supports_estimator: boolean(),
  supports_sampler: boolean(),
  supports_batch: boolean(),
  supports_dynamic: boolean(),
  supports_cancel_in_running: boolean(),
  supports_calibration_payload: boolean(),
  target_class: :gate_model | :analog | :simulator
}
```

### State Mapping Strategy

Use deterministic mapping tables in adapter modules (no dynamic string parsing outside adapter boundary).

| Normalized | IBM Runtime | AWS Braket | Azure Quantum |
| --- | --- | --- | --- |
| `:submitted` | provider submitted state | task created/submitted | job submitted |
| `:queued` | queued/pending equivalent | queued | waiting/queued equivalent |
| `:running` | running | running | executing |
| `:completed` | completed/done | completed | succeeded |
| `:cancelled` | cancelled | cancelled | cancelled |
| `:failed` | failed/error terminal | failed | failed |

Notes:

1. Preserve raw provider status in `metadata.raw_state`.
2. If mapping is unknown, return typed invalid-response error, do not coerce silently.

## Testing Strategy

Test layers:

1. Unit tests:
   - state and error mapping tables
   - capability preflight validation
   - redaction utilities
2. Contract tests (fixture-backed):
   - deterministic remote response replay for all lifecycle operations
3. Acceptance tests:
   - feature scenarios mapped to step modules
4. Optional smoke tests:
   - credentialed lanes guarded by env flags in CI
5. Observability contract tests:
   - span/metric key stability
   - profile behavior checks
   - redaction checks in logs/events

Determinism policy:

1. Fixture mode is mandatory for CI default lanes.
2. Live provider lanes are non-blocking or separately gated by release policy.
3. Fixtures must include both successful and representative failure payloads.

## Risk Register

1. Provider API drift.
   - Mitigation: version pinning + mapping tests + strict adapter boundaries.
2. Hidden nondeterminism in live provider interactions.
   - Mitigation: deterministic fixture-first testing and typed terminal state handling.
3. Scope expansion from provider-specific edge cases.
   - Mitigation: enforce v0.5 non-goals and capability preflight strategy.
4. Security leakage through diagnostics.
   - Mitigation: centralized redaction utilities and redaction tests in failure paths.
