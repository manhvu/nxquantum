# v1.0 Hybrid Quantum-AI Integration Guide

Status note (as of March 23, 2026):

1. This guide defines API usage and benchmark-integration contracts for Phase 20.
2. It is implementation guidance; production `NxQuantum.AI` modules are planned under Phase 19/20.

## Purpose

Provide a deterministic integration recipe so teams can run hybrid benchmark scenarios and compare against classical baselines using stable contract envelopes.

## Required Inputs

1. Tool contract spec:
   - `docs/v1.0-quantum-ai-tool-contracts.md`
2. Benchmark scenario spec:
   - `docs/v1.0-hybrid-quantum-ai-benchmark.md`
3. Dataset IDs/version metadata:
   - `rq_small_v1` / `rq_medium_v1`
   - `co_knapsack_v1`
   - `lf_mixed_v1`

## API Usage Patterns

## Pattern 1: Sync Execution (Reference)

Use sync for local deterministic evaluation and low-latency benchmarking.

```elixir
request = %{
  schema_version: "v1",
  request_id: "req-sync-001",
  correlation_id: "corr-sync-001",
  tool_name: "quantum_kernel_rerank.v1",
  tool_version: "v1",
  input: %{
    dataset_id: "rq_small_v1",
    query_id: "q-42",
    candidate_ids: ["d-1", "d-2", "d-3"],
    quantization: %{
      codec: :turboquant,
      mode: :prod_unbiased,
      bit_width: 4,
      seed: 20260328,
      parallel_mode: :auto,
      max_concurrency: System.schedulers_online()
    }
  },
  execution_policy: %{
    timeout_ms: 3000,
    transport_mode: :fixture,
    fallback_policy: :strict
  }
}

# Sync transport scaffold usage until production tool runner lands.
{:ok, result} =
  NxQuantum.Adapters.AIToolTransport.McpJsonRpcSync.invoke_sync(
    request,
    stub_result_envelope: %{
      schema_version: "v1",
      request_id: request.request_id,
      correlation_id: request.correlation_id,
      status: :ok,
      tool_name: request.tool_name,
      output: %{ranked_ids: ["d-2", "d-1", "d-3"]}
    }
  )
```

## Pattern 2: Async Execution (Reference)

Use async to benchmark queue/event-style operational behavior and fallback telemetry.

```elixir
request = %{
  schema_version: "v1",
  request_id: "req-async-001",
  correlation_id: "corr-async-001",
  tool_name: "constrained_optimize.v1",
  tool_version: "v1",
  input: %{dataset_id: "co_knapsack_v1", problem_id: "p-200"}
}

{:ok, dispatch} =
  NxQuantum.Adapters.AIToolTransport.CloudEventsAsync.publish_async(request, dispatch_id: "dispatch-001")

terminal = %{
  schema_version: "v1",
  request_id: "req-async-001",
  status: :ok,
  output: %{solution: [1, 0, 1, 1], objective_value: 91.0}
}

{:ok, %{status: :ok, result: ^terminal}} =
  NxQuantum.Adapters.AIToolTransport.CloudEventsAsync.fetch_async_result(
    dispatch.dispatch_id,
    stub_terminal_result: terminal
  )
```

## Benchmark Runner Contract (Planned Scripts)

## Inputs

1. `--scenario` (`rerank_quality_delta` | `constrained_optimization_assistant` | `latency_fallback_impact`)
   - TurboQuant lane: `rerank_quality_delta_turboquant`
2. `--dataset` (dataset ID from spec)
3. `--seed` (integer)
4. `--runtime-profile`
5. `--transport-mode` (`fixture` | `live_smoke` | `live`)

Transport mode semantics:

1. `fixture`: deterministic baseline lane for reproducible checks.
2. `live_smoke`: credential-gated readiness lane (env/config and contract-path validation) with fixture-compatible payload semantics by default.
3. `live`: credential-gated production lane for real remote provider lifecycle execution.

## Outputs

1. JSON report in `tmp/hybrid_bench/*.json`
2. Markdown summary in `docs/v1.0-hybrid-quantum-ai-benchmark-report-YYYY-MM-DD.md`
3. Comparison deltas vs classical baseline.
4. TurboQuant lane metrics include `memory_bytes_per_vector` and compression ratio evidence.

## Baseline Discipline

1. Every scenario must have a classical-only baseline with same dataset split and seed.
2. Hybrid claims must include absolute and relative deltas.
3. If fallback is allowed, result must include fallback ratio and impact on quality/latency.

## Troubleshooting Expectations

For each benchmark run persist:

1. `request_id`, `correlation_id`, `scenario_id`, `dataset_id`
2. typed transport/tool error codes,
3. resolved runtime profile and transport mode,
4. fallback decision metadata.

## Adoption Sequence (Recommended)

1. Start with MCP sync adapter for deterministic contract checks.
2. Add CloudEvents async adapter path for operational rollout testing.
3. Integrate broker-backed async adapter implementation under the same port.
4. Promote only when benchmark gates are reproducible and documented.
