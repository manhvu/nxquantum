# v0.8 Provider Live Execution Plan

## Goal

Deliver authenticated live execution on IBM Runtime, AWS Braket, and Azure Quantum while preserving fixture-first deterministic regression behavior.

## Transport Contract

Supported provider bridge transport modes:

1. `:fixture` (default deterministic lane)
2. `:live_smoke` (credential-gated readiness lane; validates env/config/contract parity without requiring full remote lifecycle execution)
3. `:live` (credential-gated production lane with remote lifecycle execution)

Current implementation note (as of March 25, 2026):

1. `:live_smoke` readiness is env/config-gated and exposed in transport metadata.
2. Provider adapters keep fixture-style lifecycle payload semantics in `:live_smoke` mode.
3. `:live` mode is the lane that executes remote provider lifecycle calls.

## Required Envelope Parity

All lanes return stable typed envelopes with:

1. `schema_version`
2. `request_id`
3. `correlation_id`
4. `idempotency_key`

## Delivery Notes

1. Provider-specific transport details remain in adapters.
2. Domain/application contracts remain provider-agnostic and typed.
3. Live runs can be captured into replay fixtures with provenance metadata.

## CI Live-Smoke Configuration (Core Provider Set)

Enable `live_smoke` lanes with either the global gate or provider-specific gate:

1. global: `NXQ_PROVIDER_LIVE_SMOKE=1`
2. provider-specific:
   - `NXQ_PROVIDER_LIVE_SMOKE_IBM_RUNTIME=1`
   - `NXQ_PROVIDER_LIVE_SMOKE_AWS_BRAKET=1`
   - `NXQ_PROVIDER_LIVE_SMOKE_AZURE_QUANTUM=1`
   - `NXQ_PROVIDER_LIVE_SMOKE_GOOGLE_QUANTUM_AI=1`

Google `live_smoke` readiness requires `provider_config` with:

1. `auth_token`
2. `project_id`
3. `location`
4. `processor_id`

Note:

1. `live_smoke` validates env/config and contract-path readiness while keeping fixture-compatible payload semantics.
2. Full remote lifecycle execution remains `transport_mode: :live`.
