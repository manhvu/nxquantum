# v0.5 Provider Support Tiers and Known Limits

Status note (as of March 20, 2026):

1. This document defines support-tier policy for the core provider set.
2. Tier policy is used for release-note framing and migration risk visibility.
3. Core provider set catalog source of truth: `docs/provider-support-catalog.md`.

## Tier Definitions

1. `stable`
   - Acceptance-covered and fixture-verified in default release gates.
2. `beta`
   - Usable with documented caveats; behavior is typed but may evolve.
3. `experimental`
   - Explicitly non-default paths with limited compatibility guarantees.

## Provider Tier Matrix (v0.5)

| Provider adapter | Tier | Scope summary | Known limits |
| --- | --- | --- | --- |
| `NxQuantum.Adapters.Providers.IBMRuntime` | stable | Estimator/sampler lifecycle (`submit/poll/cancel/fetch_result`) with typed normalization | Depends on explicit backend/channel config |
| `NxQuantum.Adapters.Providers.AwsBraket` | stable | Gate-model lifecycle with capability preflight and typed state/error mapping | Non-gate-model workflows are out of v0.5 primary scope |
| `NxQuantum.Adapters.Providers.AzureQuantum` | beta | Workspace/target/provider lifecycle with caveat metadata | Cancellation semantics can vary by provider phase |
| `NxQuantum.Adapters.Providers.GoogleQuantumAI` | stable | Estimator/sampler lifecycle (`submit/poll/cancel/fetch_result`) with typed normalization | Requires explicit project/location/processor configuration |
