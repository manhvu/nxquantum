## Feature Acceptance Criteria

### F1) Provider Capability Contract v1

Required outcomes:

1. Capability envelope is explicit and typed for all top-3 providers.
2. Unsupported capability requests fail before remote submission.
3. Error payloads include provider and target context.
4. Capability contract schema is stable and versioned.

Acceptance checklist:

1. `supports_estimator`, `supports_sampler`, `supports_batch`, `supports_dynamic`, `supports_cancel_in_running`, `supports_calibration_payload`, `target_class` are present.
2. Unknown capability or malformed capability contract returns typed diagnostics.
3. Capability checks produce deterministic results for equivalent inputs.

### F2) IBM Runtime Provider Adapter

Required outcomes:

1. IBM lifecycle supports typed `submit/poll/cancel/fetch_result`.
2. IBM statuses map deterministically to normalized NxQuantum states.
3. Primitive-centric workflows (`estimator`, `sampler`) are supported by contract.
4. Auth/config/backend failures are mapped to typed error codes.

Acceptance checklist:

1. Happy path lifecycle completes with normalized result envelope.
2. Cancellation path returns typed terminal semantics.
3. Invalid backend/capability and auth errors return stable codes and metadata.

### F3) AWS Braket Provider Adapter

Required outcomes:

1. Braket task lifecycle is normalized to NxQuantum state model.
2. Target/device constraints are enforced via capability preflight.
3. Gate-model workflow contract is explicit for v0.5 scope.
4. Failed terminal states include typed reasons and provider context.

Acceptance checklist:

1. Happy path terminal result maps cleanly.
2. Unsupported target class is rejected with typed capability mismatch.
3. Terminal failure and cancellation states are deterministic and typed.

### F4) Azure Quantum Provider Adapter

Required outcomes:

1. Azure workspace/target/provider selection is validated via typed config contract.
2. Lifecycle operations normalize to NxQuantum state model.
3. Cancellation caveats are surfaced explicitly in metadata.
4. Target/provider mismatch returns typed capability error.

Acceptance checklist:

1. Happy path lifecycle yields normalized terminal result.
2. Cancellation behavior includes caveat metadata where relevant.
3. Mismatch failures are deterministic and typed.

### F5) Cross-Provider Contract Equivalence

Required outcomes:

1. Equivalent workflows return consistent normalized envelope structure across providers.
2. Non-portable behavior is explicit under metadata (not hidden).
3. No silent fallback to alternate provider occurs.
4. Cross-provider error taxonomy is stable.

Acceptance checklist:

1. Shared fields (`id`, `state`, `provider`, `target`, `payload`, `metadata`) are present in all providers.
2. Provider-specific extensions are namespaced under metadata.
3. Unknown provider status maps to typed invalid-response error (never silently coerced).

### F6) Migration and Enablement Assets

Required outcomes:

1. Provider-specific migration packs exist for IBM Runtime, Braket, and Azure workflows.
2. Tutorials are runnable via deterministic fixture lanes even without cloud credentials.
3. Claims in migration docs are traceable to acceptance scenarios or benchmark evidence.

Acceptance checklist:

1. Migration playbook includes explicit preflight checklist and parity checkpoints.
2. Every migration path has at least one mapped acceptance scenario.
3. Known provider caveats and support tiers are documented.

### F7) OpenTelemetry Observability Standard

Required outcomes:

1. Traces/logs/metrics are standardized for all top-3 provider lifecycles.
2. Visibility profiles are configurable (`high_level`, `granular`, `forensics`).
3. Redaction and cardinality safeguards are enforced.
4. Experiment fingerprint and portability-delta telemetry contracts are deterministic.

Acceptance checklist:

1. Mandatory span names and attributes are emitted for provider lifecycle.
2. Core latency/error metrics exist with stable names/types.
3. High-level profile avoids high-cardinality noise by default.
4. Fingerprint output is deterministic for equal canonicalized inputs.
5. Portability-delta telemetry is emitted for comparable cross-provider workloads.
