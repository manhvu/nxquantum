# NxQuantum v0.3 Feature Specification

## Release Theme

Hardware-ready QML primitives with batched execution and noise-aware result pipelines.

## Why v0.3 Matters

1. Make NxQuantum practical for real hybrid training loops by removing per-sample circuit bottlenecks.
2. Align the API with modern hardware abstractions (`Estimator`/`Sampler`) rather than low-level raw outputs.
3. Prepare compilation/transpilation boundaries so hardware adapters can be added without API rewrites.

## Priority Selection (Pareto)

Selected for v0.3 (highest community value per delivery risk):

1. **P0: Primitives API hardening** (`Estimator` + `Sampler`).
2. **P0: Batched parameterized circuit execution** (`batch` first-class).
3. **P1: Pluggable error mitigation pipeline** (readout + ZNE baseline).
4. **P1: Topology-aware transpilation interface** (metadata + baseline pass behavior).
5. **P2: Dynamic-circuit IR foundation** (classical register + branch nodes, no full execution yet).

Deferred post-v0.3:

1. Full dynamic-circuit simulation execution in `Nx.Defn` kernels.
2. Tensor-network/MPS simulator fallback.

## Feature Set

### F1. Primitives API (Estimator + Sampler) [P0]

Goal:

1. Expose stable hardware-agnostic primitives for expectation and sampling workflows.

Scope:

1. `NxQuantum.Estimator` contract for expectation values over observables/Hamiltonians.
2. `NxQuantum.Sampler` contract for probabilities and shot-based samples.
3. Deterministic result typing (`{:ok, result}`/typed error metadata).
4. Multi-observable batching support.

Acceptance:

1. Estimator and sampler return deterministic outputs for same seed/input.
2. Observable list execution is shape-stable and backend-agnostic.
3. Typed errors are stable for invalid observables and option schemas.

### F2. Batched PQCs [P0]

Goal:

1. Treat parameter batches as a core execution dimension.

Scope:

1. Batch-aware parameter binding for circuit evaluation.
2. Batched expectation and batched sampling APIs.
3. Baseline vectorization strategy (broadcast/vectorize/defn-friendly kernels).
4. Deterministic behavior across batch sizes.

Acceptance:

1. Batched outputs match scalar-loop references within tolerance.
2. Batch shape contract is explicit (`{batch, ...}`).
3. `batch=1` behavior is equivalent to scalar API.

### F3. Pluggable Error Mitigation Pipeline [P1]

Goal:

1. Make noisy outputs usable by default in QML pipelines.

Scope:

1. Mitigation pipeline behavior and composition API.
2. Built-in readout mitigation pass (calibration matrix).
3. Built-in simple ZNE pass (linear extrapolation baseline).
4. Deterministic seed + mitigation contract.

Acceptance:

1. Pipeline composition order is explicit and deterministic.
2. Mitigated outputs are reproducible for same input/calibration/seed.
3. Pipeline works with both estimator and sampler outputs.

### F4. Topology-Aware Transpilation Interface [P1]

Goal:

1. Establish the compiler boundary required for real hardware constraints.

Scope:

1. Circuit metadata for topology and coupling map.
2. Transpiler behavior (`@callback`) and pass registry.
3. Deterministic routing pass: shortest-path SWAP insertion for non-native two-qubit interactions.
4. Deterministic tie-break policy for equal-length paths (`lexicographic_path`).
5. Optional mapping policy: `restore_mapping: true | false`.
6. Transpile report metadata (added gates, depth delta, mapping summary, routed edges).

Acceptance:

1. Non-native coupling requests return typed transpilation errors in strict mode.
2. Insert-swaps mode selects deterministic shortest paths for known coupling maps.
3. Equal-length path cases are resolved by explicit deterministic tie-break rule.
4. Transpiled circuit preserves expectation within tolerance.
5. Report contains deterministic diagnostics for audits (`routed_edges`, `logical_to_physical_map`, `topology_id`).

Step 1 implementation target (current highest-priority limitation):

1. Replace placeholder SWAP insertion with deterministic shortest-path routing for `:coupling_map` and `:heavy_hex`.
2. Ship report fields needed for debugging/reproducibility:
   - `added_swap_gates`
   - `depth_delta`
   - `routed_edges`
   - `logical_to_physical_map`
   - `topology_id`
3. Keep dynamic remapping policy explicit and deterministic via `restore_mapping`.

### F5. Dynamic Circuit IR Foundation [P2]

Goal:

1. Prepare domain model for mid-circuit measurement and feed-forward control.

Scope:

1. IR nodes for classical register writes and conditional gate blocks.
2. Validation rules for branch structure and classical references.
3. Serialization/inspection support for dynamic IR nodes.
4. Explicit limitation: no production execution engine in v0.3.

Acceptance:

1. IR validates legal classical-control graph structures.
2. Invalid branch/control references return typed errors.
3. Feature is opt-in and clearly marked experimental.

## Proposed API Additions (v0.3)

```elixir
alias NxQuantum.Circuit
alias NxQuantum.Estimator
alias NxQuantum.Sampler
alias NxQuantum.Mitigation
alias NxQuantum.Transpiler

# Batched Estimator
{:ok, %Estimator.Result{values: values}} =
  Estimator.run(circuit, observables: [:pauli_z], params: batched_theta, shots: 1_000, seed: 7)

# Sampler + mitigation pipeline
{:ok, sample_result} =
  Sampler.run(circuit, shots: 4_096, seed: 7)

{:ok, mitigated} =
  sample_result
  |> Mitigation.pipeline([
    {:readout, calibration: readout_calibration},
    {:zne_linear, scales: [1.0, 2.0, 3.0]}
  ])

# Topology-aware transpilation
{:ok, transpiled, report} =
  Transpiler.run(circuit,
    topology: {:heavy_hex, coupling_map},
    mode: :insert_swaps
  )
```

## Architecture and DDD Boundaries

1. Domain:
   - `Circuit` enriched with optional topology/dynamic metadata.
   - Primitive result structs as value objects.
2. Application:
   - `Estimator.run/2`, `Sampler.run/2`, `Transpiler.run/2`, `Mitigation.pipeline/2`.
3. Ports:
   - simulator, backend, transpiler-pass behaviors.
4. Adapters:
   - state-vector + backend-specific execution tuning.

## Delivery Plan

### Milestone A (P0): Primitives + Batch Core

1. Finalize estimator/sampler contracts and result structs.
2. Implement batched parameter path and deterministic shape checks.
3. Add acceptance + unit/property coverage.

### Milestone B (P1): Mitigation + Topology Interface

1. Implement mitigation pipeline behavior and built-in passes.
2. Add transpiler behavior and deterministic shortest-path routing pass.
3. Add typed reports/errors and semantic equivalence checks.

### Milestone C (P2): Dynamic IR Foundation + Release Hardening

1. Add dynamic IR node model and validators.
2. Document experimental status and boundaries.
3. CI/docs/bench polish for v0.3 release readiness.

## Success Metrics

1. Batched execution: measurable speedup over scalar loop for `batch >= 32`.
2. Determinism: repeated seeded runs are byte-identical in acceptance tests.
3. DX: contributors can implement each feature via feature-to-step mapping without touching unrelated modules.

## Milestone E (v0.3 Foundation) Evidence Checklist

Required executable evidence for v0.3 foundation completion:

1. Stable primitive contracts (`Estimator`, `Sampler`) are validated by deterministic contract tests.
2. Batched estimator/sampler APIs are implemented and validated for deterministic behavior and batch-shape errors.
3. Mitigation and topology-transpilation feature scenarios execute with deterministic diagnostics.
4. Dynamic IR foundation has module-level validation and explicit `dynamic_execution_not_supported` runtime boundary.
5. Feature-to-step mappings are published and aligned with the current implementation.

## Risks and Mitigations

1. Risk: API sprawl across many new modules.
   Mitigation: keep stable facade modules and typed result structs.
2. Risk: backend-specific behavior drift.
   Mitigation: shared deterministic references and backend-tolerance contracts.
3. Risk: transpilation complexity expanding scope.
   Mitigation: strict v0.3 boundary with baseline deterministic passes only.
